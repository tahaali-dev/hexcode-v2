Our Audit Story — What We Found When We Opened the Codebase

When our team first opened the ZiShop codebase, the overall structure told a familiar story: a motivated developer who moved fast, shipped features, and deferred hardening for later. The architecture is sound in its intent — a Next.js storefront connected to Sanity CMS with Redux for state. But as we peeled back each layer, we found that “later” had arrived before the foundations were ready.

The most immediate concern was not a styling inconsistency or a performance gap — it was a fundamental authentication bypass. A single error condition on the sign-up flow, if triggered deliberately, grants any visitor full authenticated access without a real account. Alongside this, the Sanity CMS write token is embedded in every page’s JavaScript bundle, visible to any curious customer who opens DevTools.

Moving into the commerce layer, we discovered the checkout flow does not exist. “Place Order” routes customers back to the homepage. The cart never clears after checkout. The search bar accepts input but returns nothing. These are not minor gaps — they are the core transactions the platform exists to perform.

None of this is irreversible. Every finding maps to a clear strategic path forward. The PRD migration to Next.js 14, Auth.js, PostgreSQL with Prisma, and Zustand addresses the majority of architectural concerns at the root. This report prioritises the path from here to a production-ready platform.

Critical Issues

6 issues
Critical
Authentication Bypass on Registration Failure
SecurityPRD: Auth.js (NextAuth v5) required
▼
⚠
Website Impact: Any visitor can gain authenticated access by deliberately triggering a server error during sign-up — bypassing all login checks and gaining full user privileges without a real account. No credential is needed.

Problem

When the registration API returns a 500 status, the front-end silently logs the user in using unverified data. A malicious actor can intentionally trigger this error condition and gain authenticated status with arbitrary credentials they invented.

Our Recommendation

Remove the error-path login fallback entirely. Authentication must only succeed via a verified server response. Migrate to Auth.js session handling where session state is established server-side and cannot be influenced by client-side error branches.

Critical
Sanity Write Token Exposed in Browser Bundle
SecurityPRD: TypeScript Strict + secure env var handling
▼
⚠
Website Impact: The CMS write token is embedded in every page’s JavaScript bundle. Any visitor who opens DevTools can extract it and use it to read, modify, or delete all CMS content — products, configurations, and stored user data.

Problem

The NEXT_PUBLIC_ prefix forces Next.js to include this variable in the client-side bundle at build time. This design-level misconfiguration makes the secret permanently public for every deployed build and cannot be patched without a variable rename and full redeployment.

Our Recommendation

Rename the variable to remove the NEXT_PUBLIC_ prefix immediately and move all Sanity write operations to server-only API routes. Client components must receive read-only content through server-rendered props, never through a privileged write token.

Critical
Cart Never Clears — Persistent Ghost Cart Bug
Cart / CheckoutPRD: Reliable cart and checkout state required
▼
⚠
Website Impact: After checkout or manual clear, the cart retains all items permanently. Customers see stale items from previous sessions, leading to incorrect order totals and a completely broken post-purchase experience that destroys trust.

Problem

The clearCart reducer uses an Immer no-op pattern. Reassigning the local state parameter has no effect because Immer only responds to direct mutations or explicit return values. The same bug exists in both the cart and favourites slices simultaneously.

Our Recommendation

Refactor both reducers to return the initial state object directly instead of reassigning the parameter. As part of the PRD migration to Zustand, replace the Redux cart slice with a Zustand store that handles persistence and reset correctly from the outset.

Critical
Order Button Links to Homepage — No Checkout Flow Exists
Cart / CheckoutPRD: Stripe with webhook-confirmed orders required
▼
⚠
Website Impact: Clicking “Place Order” returns customers to the homepage. There is no payment step, no order confirmation, and no actual transaction — the platform cannot generate any revenue in its current state.

Problem

The order summary component routes to “/” on click rather than a checkout route. No checkout page, Stripe integration, or order confirmation flow has been implemented, making the entire commerce functionality non-operational for every visitor.

Our Recommendation

Build a dedicated checkout flow with a Stripe Payment Intent API route, a client-side payment form, and a webhook handler to confirm orders server-side. The PRD explicitly requires webhook-confirmed payment status before any order is considered complete.

Critical
API Key Hardcoded in Source Code
SecurityPRD: Production-ready secret management required
▼
⚠
Website Impact: A live third-party API key committed in source code is permanently in git history. Anyone with repository access can extract it and incur costs or access rate-limited data on behalf of the business. The live exchange rate fetch is also disabled, meaning customers see a permanently stale currency rate.

Problem

The ExchangeRate API key is embedded as a plain string constant in the application code. Even if removed from the latest commit, it remains in git history indefinitely. The disabled fetch means currency conversion uses a hardcoded fallback that will never reflect real values.

Our Recommendation

Rotate the exposed API key immediately through the service provider’s dashboard. Move all secrets to server-only environment variables. Restore the live fetch through a server-side API route that proxies the request without exposing the key to client-side code.

Critical
Open Redirect Vulnerability on Sign-Up
SecurityPRD: Auth.js with validated session redirects
▼
⚠
Website Impact: Attackers can craft sign-up URLs that redirect users to malicious external sites after registration, enabling phishing campaigns where customers believe they are on the legitimate site throughout the entire process.

Problem

The redirect query parameter is used directly in router.push() without any validation that the destination is an internal path. Any external URL supplied in this parameter will be followed automatically after successful sign-up.

Our Recommendation

Validate all redirect targets against a strict allowlist of internal paths before use. Discard any value that does not match a known internal route and replace it with the default home path. Auth.js handles this validation natively as part of its session management.

High
JWT Tokens Valid for 30 Days With No Revocation
SecurityPRD: Auth.js server-side session management
▼
⚡
Website Impact: If a user’s account is compromised, there is no way to invalidate their session. The attacker retains full authenticated access for up to 30 days even after the legitimate user changes their password or reports the breach.

Problem

Long-lived JWTs with no server-side revocation create an attack window that cannot be closed without rotating the signing secret and logging out all users simultaneously. There is also no logout endpoint that actually invalidates the token.

Our Recommendation

Migrate to Auth.js which provides server-managed sessions with database-backed revocation. In the interim, reduce token expiry significantly and implement a server-side token denylist checked on each authenticated request.

High
No Input Validation on Authentication Endpoints
SecurityPRD: TypeScript Strict + production-grade API security
▼
⚡
Website Impact: Malformed or malicious data submitted to login and registration causes server errors and information disclosure. Without rate limiting, automated bots can hammer these endpoints continuously, potentially locking out real users or brute-forcing credentials.

Problem

Authentication handlers read directly from req.body without validating format, length, or type. There is no schema validation, no sanitisation, and no rate limiting — all three absent from every authentication endpoint simultaneously.

Our Recommendation

Introduce a Zod validation layer at every API boundary with strict schemas for all authentication inputs. Add rate limiting middleware on authentication routes to prevent brute-force and credential-stuffing attacks before they can impact real users.

High
GROQ Queries Built With String Interpolation
DatabasePRD: Sanity CMS with parameterised queries
▼
⚡
Website Impact: User-controlled URL segments embedded in CMS queries allow crafted requests that return data from unrelated document types — including unpublished draft content, internal configurations, or private records not intended for display.

Problem

Product detail pages build GROQ queries by interpolating URL parameters directly into the query string. GROQ is read-only so database writes are impossible, but unauthorised data exfiltration across document types is a genuine exposure.

Our Recommendation

Replace all string-interpolated queries with the parameterised form of client.fetch(), passing variables as a separate object. This separates query structure from user-supplied values and closes the interpolation attack surface on every product detail page.

High
Homepage Fetches All Products With All Fields — No Projection
PerformancePRD: Next.js 14 optimised data fetching patterns
▼
⚡
Website Impact: The homepage downloads the entire product catalogue including every field on each visit where cache is stale. On a 100+ product catalogue this causes multi-second page loads that directly increase bounce rates and reduce first-impression conversions.

Problem

The getStaticProps query fetches every field from every product without a GROQ projection. There is also no revalidate interval, so Incremental Static Regeneration is completely disabled and the page serves indefinitely stale content after the first build.

Our Recommendation

Add a GROQ projection selecting only fields required for the listing view. Set a revalidate interval matching content update frequency. For the PRD migration, App Router with React Server Components and Next.js fetch cache provides automatic, granular cache control without manual ISR configuration.

High
No HTTP Security Headers Configured
SecurityPRD: Production-ready security posture required
▼
⚡
Website Impact: Without security headers, the site can be embedded in a hidden iframe for clickjacking attacks and has no Content Security Policy to block injected scripts — leaving every page open to cross-site scripting exploits.

Problem

The Next.js configuration exports no headers() function. X-Frame-Options, Content-Security-Policy, X-Content-Type-Options, Referrer-Policy, and Permissions-Policy are entirely absent from every server response.

Add a headers() export to next.config.js with a complete security header set. Define a Content Security Policy whitelisting only trusted script and style sources. Use a centralised header management approach to test and maintain these configurations systematically.

High

FrontendPRD: Performance-first Next.js 14 architecture
▼
⚡
Website Impact: The homepage briefly shows no products on every page load before data appears. This jarring flash signals a broken page and increases the likelihood of visitors leaving before the product grid becomes visible.

Problem

Server-fetched product data is passed as props but dispatched into Redux via useEffect. The component renders once empty then re-renders after the effect fires — a double render entirely avoidable by using the props directly in the render path.

Remove the useEffect dispatch and consume server props directly in the component render. For the PRD migration, App Router Server Components eliminate this pattern entirely — data is fetched and rendered server-side with no client-side re-hydration step.

▼

Website Impact: If a required environment variable is missing in a deployment, the application starts successfully and fails at runtime when the variable is first accessed — often in a customer-facing code path, causing a live production outage that is hard to diagnose quickly.

Problem

There is no startup validation checking that required environment variables are present and correctly formatted before the application begins serving requests. Missing variables only surface when the specific code path using them is hit in production traffic.

Our Recommendation

Add a Zod-based startup validation step that checks all required environment variables against a defined schema before the server accepts any requests. Provide clear error messages listing exactly which variables are missing or malformed, making deployment failures obvious and fast to resolve.
